# Sequence-Structure Alignment Analysis

## 1. Problem statement

ProteinTraitsMech deliberately separates records by representation axis: sequence signatures and sequence-local features live on `SEQUENCE`; 3-D folds, structural domains, active sites, binding sites, metal sites, and secondary-structure features live on `STRUCTURE`; a small `SEQUENCE_STRUCTURE` axis is reserved for traits whose definition already depends on both sequence and structure. This is enforced in the LinkML schema by the `trait_axis` slot and category-prefix rules for `SEQ_*`, `STRUCT_*`, `MIXED_*`, `FUNC_*`, and `EVO_*` in `src/proteintraitsmech/schema/proteintraitsmech.yaml:308-320` and `src/proteintraitsmech/schema/proteintraitsmech.yaml:409-465`.

The split keeps provenance clean but leaves a missing relationship layer: a sequence-axis kinase motif and a structure-axis kinase active site can describe the same physical protein region without any asserted connection between the records. The right framing is not "merge the records"; it is "project both annotations onto the same exemplar protein sequence, then assert typed cross-axis edges when their residue sets overlap."

The merge rules forbid collapsing such pairs. `.claude/skills/merge-within-axis/SKILL.md:32-34` states that comparison is same-axis and same-category only, and `.claude/skills/merge-traits/SKILL.md:42-48` applies the same guard to non-exact identity rules. Cross-axis sequence-structure reconciliation must therefore create `trait_relations`-style or overlay TSV edges, never a merged SEQUENCE+STRUCTURE record.

Verified correction to the prompt: `CanonicalExample` is not coordinate-free in the current tree. It has top-level exemplar metadata (`protein_id`, `protein_label`, `taxon_id`, `taxon_label`, `sequence_length`, `sequence`) and also optional `features`, whose class `SequenceFeatureAnnotation` carries `start` and `end` coordinates (`src/proteintraitsmech/schema/proteintraitsmech.yaml:515-640`). The actual gap is narrower but still decisive: there is no normalized, source-specific "this trait occurrence on this exemplar spans these residues in this declared coordinate frame" object. Existing `features` are UniProt feature tracks attached to an example protein, not a guaranteed location of the record's own signature or structure-derived occurrence.

## 2. Coordinate-frame model

Use the UniProt canonical protein sequence as the shared coordinate frame wherever possible. For each candidate pair, the builder should require the same `CanonicalExample.protein_id` or an explicit structure-to-UniProt residue map, convert every annotation into either an interval `[start,end]` or a residue set `{p1,p2,...}`, and compute overlap on 1-indexed UniProt residues. AlphaFold/TED identifiers are comparatively easy because AlphaFold models are accession-indexed; PDB-derived sources need SIFTS-style chain/residue mapping because PDB author and label residue numbering can differ from UniProt numbering.

| Axis | Category | Source examples verified in tree | Positions available now? | Native frame | Class vs instance |
|---|---|---|---|---|---|
| SEQUENCE | `SEQ_MOTIF` | PROSITE pattern records such as `PROSITE:PS00107` in `data/traits/sequence/pattern/protein-kinase-atp.yaml` | `sequence_pattern` is stored; exemplar `features` may be stored, but the PROSITE match coordinates are not explicitly marked as the record's occurrence | Pattern is class-level; computed hits are UniProt sequence coordinates | Class signature, instance coordinates computed per protein |
| SEQUENCE | `SEQ_DOMAIN` / `SEQ_HOMOLOGOUS_SUPERFAMILY` | Pfam `PF00069` and InterPro `IPR000719` records | Class records generally store signatures/xrefs, not per-example domain intervals; UniProt/InterPro APIs can supply match start-end | UniProt sequence for InterPro/Pfam matches | Class signature; hits are instance-level |
| SEQUENCE | `SEQ_ACTIVE_SITE`, `SEQ_BINDING_SITE`, `SEQ_PTM_SITE`, PTM subtypes | Schema categories in `ProteinTraitCategoryEnum`; seeders route ELM/PROSITE/UniProt-like records into sequence categories | UniProt FT/ELM-style examples can carry `features.start/end`; class records may only store regex/motif or ontology anchor | UniProt sequence | Site class plus per-protein occurrence |
| STRUCTURE | `STRUCT_ACTIVE_SITE` | M-CSA, e.g. `MCSA:757` in `data/traits/structure/active_site/mcsa/camp-dependent-protein-kinase-mcsa757.yaml` | Current record has UniProt exemplar sequence and feature coordinates populated; seeder originally emits `canonical_examples` from reference UniProt IDs (`scripts/seed_mcsa.py:194-255`) | In current examples, UniProt sequence coordinates; raw source also has PDB/CATH/UniProt xrefs | Structure/mechanism class with per-reference occurrence |

The current browser already has a useful visual precedent. `scripts/build_docs_index.py:302-321` projects `CanonicalExample.features` into compact `feats` tuples, and `docs/browse.js:903-980` renders per-residue colored strips by axis. That is a display model for overlapping tracks, not yet a relation-building model.

## 3. Overlap -> edge semantics

Cross-axis overlap produces relationship edges only. The existing relation shape is `TraitRelation` with a `predicate`, `object`, and `relation_source`, where predicates must be `biolink:` or `RO:` CURIEs (`src/proteintraitsmech/schema/proteintraitsmech.yaml:790-813`). Existing overlays use TSVs with `subject`, `predicate`, `object`, `relation_source` and are loaded bidirectionally into the browser's `eq` projection by `scripts/build_docs_index.py:256-278`.

| Coordinate relationship on same UniProt protein | Suggested predicate | Threshold | Meaning | Merge? |
|---|---|---|---|---|
| Same residue set, especially single-residue catalytic or binding sites | `biolink:related_to` with `relation_source=seq-struct-same-residue`, or a curated RO predicate if selected | Exact residue-set equality | Same physical residue/site seen through two representations | Never |
| One feature contained in another, e.g. catalytic motif inside domain/fold | `biolink:part_of` from smaller/local feature to larger region, or `biolink:narrow_match` for class containment | `|A∩B|/|A| = 1.0` and `|A| < |B|` for strict containment | Smaller physical feature lies inside larger structural/sequence region | Never |
| Strong interval agreement between two local regions | `biolink:overlaps` or `biolink:related_to` with high-confidence source tag | reciprocal overlap `min(|A∩B|/|A|, |A∩B|/|B|) >= 0.80` | Same region candidate across axes | Never |
| Partial overlap | `biolink:overlaps` | Jaccard `>= 0.20` or min-overlap `>= 0.20`, source-specific | Biologically related but not enough for same-region claim | Never |
| Same exemplar protein but disjoint coordinates | no edge by default | `|A∩B| = 0` | Co-occurrence only | Never |

For sites, exact residue identity should dominate interval statistics. For domains/folds, use reciprocal overlap and Jaccard; for singleton sites inside domains, use containment rather than "same feature." This mirrors the localized-feature warning in `scripts/build_member_overlap.py:22-34` and the Tier-2 region verification in `scripts/verify_region_overlap.py:1-28`.

## 4. Ranked solution options

| Rank | Approach | What it does | Schema impact | Seeder/builder impact | Data dependencies | Coverage | Browser implication | Fit to axis/cross-axis rules |
|---:|---|---|---|---|---|---|---|---|
| 1 | Overlay-only pilot (`build_sequence_structure_alignment.py`) | Read SEQ and STRUCT records, find shared `canonical_examples.protein_id`, compute/source coordinates, emit `data/equivalence/seq_struct_alignment.tsv` | None | New sibling builder to `build_equivalence.py`, `build_member_overlap.py`, `verify_region_overlap.py`, `build_structural_equivalence.py` | UniProt examples already in records; InterPro/UniProt match APIs; SIFTS for PDB records; optional regex matching for PROSITE/ELM | Best smallest useful step; works on examples with stored sequence/features and on API-resolvable signatures | Browser can show new cross-axis edges through the existing `eq` loader; detail view can already render per-example feature tracks | Strong fit: edges only, no merge |
| 2 | Schema-first coordinate-bearing occurrence fields | Add an occurrence object under `canonical_examples`, e.g. `trait_occurrences: [{source_trait_id, coordinate_frame, start, end, residue_positions, evidence_source, mapping_method}]` | Moderate, but clarifies the model | Seeders/fetchers must populate occurrence coordinates rather than generic feature tracks only | Same as overlay pilot, but persisted | Better reproducibility and auditability | Detail view can distinguish "record occurrence" from background UniProt features | Strong fit if used only to generate `trait_relations`/overlays |
| 3 | Shared aligned-example registry | Introduce reference-protein objects with one sequence frame and multiple feature tracks pointing back to trait records | Larger schema addition | New registry builder, de-duplication of repeated sequences/features across records | UniProt, InterPro, SIFTS, PDB/AlphaFold mappings; track provenance | Highest for curated exemplars and repeated browser use | Best browser model: a ProtVista/Nightingale-like track stack for SEQ vs STRUCT features | Strong but heavier than current corpus architecture |
| 4 | Structure-superposition complement | Use Foldseek/TM-align/TM-score to align folds where direct residue correspondence is weak | None or sidecar manifest extension | Extend structural-equivalence machinery; use as a supplement to residue-frame overlap | Representative structures, chain/domain chopping, Foldseek/TM-align | Good for fold correspondence across divergent proteins; poor for exact active-site equivalence without residue mapping | Could display structural-equivalence context, not per-residue overlap unless mapped back | Fit is limited to STRUCTURE evidence; should nominate, not assert cross-axis same-site |

Recommendation: start with the overlay-only pilot. It matches the existing overlay contract (`data/equivalence/*.tsv`), can reuse the browser's current edge loader, and avoids a schema migration while testing whether enough records can be placed on a shared UniProt frame. The first implementation should support three coordinate providers: stored `CanonicalExample.features`, PROSITE/ELM regex hits against stored exemplar sequences, and InterPro/Pfam match coordinates from source APIs. Add SIFTS as the next provider for BioLiP2, PDB-derived CATH/ECOD/SCOPe representatives, and raw M-CSA/PDB catalytic residues.

Incremental path:

1. Build `seq_struct_alignment.tsv` for records sharing exact `protein_id`, using only already-stored `sequence`, `features`, `sequence_pattern`, and `structural_geometry_representations`.
2. Add source fetchers for InterPro/Pfam/PROSITE match locations and SIFTS-backed PDB residue mapping; mark every edge's `relation_source` with the provider and confidence tier.
3. Promote recurring, useful computed coordinates into a schema-level occurrence object so future builds are reproducible without re-querying external APIs.
4. Add a shared aligned-example view in the browser once the edge builder has enough high-value examples.

## 5. Worked example

Example protein: UniProtKB `P17707`, S-adenosylmethionine decarboxylase proenzyme, is explicitly present on both a SEQUENCE-axis record and a STRUCTURE-axis record. The sequence-side record is `PROSITE:PS01336`, `S-adenosylmethionine decarboxylase signature`, with `trait_axis: SEQUENCE`, `trait_category: SEQ_MOTIF`, the pattern `[SA]-[FY]-[LIV]-L-[STN]-E-S-S-[LIVMF]-F-[LIV].`, and a `P17707` canonical example (`data/traits/sequence/pattern/adometdc.yaml:1-24`). The structure-side record is `MCSA:575`, `adenosylmethionine decarboxylase (eukaryotic)`, with `trait_axis: STRUCTURE`, `trait_category: STRUCT_ACTIVE_SITE`, xrefs to `PDB:1jen`, `CATH:3.30.360.50`, `CATH:3.60.90.10`, and `UniProtKB:P17707` (`data/traits/structure/active_site/mcsa/adenosylmethionine-decarboxylase-eukaryotic-mcsa575.yaml:1-22`).

Both records store the same `P17707` sequence and UniProt-derived feature track. The structure-side feature coordinates include active-site residues 8, 11, 68, 82, 229, and 243; binding residues 7, 67, 223, and 247; a cleavage/binding site 67-68; and a sequence-axis modified residue at 68 (`data/traits/structure/active_site/mcsa/adenosylmethionine-decarboxylase-eukaryotic-mcsa575.yaml:27-111`). The sequence-side record carries the same feature track on its canonical example (`data/traits/sequence/pattern/adometdc.yaml:41-122`). The CATH structural superfamily records referenced by M-CSA exist as `CATH:3.30.360.50` with representative `PDB:3ep6` and `CATH:3.60.90.10` with representative `PDB:5tvo` (`data/traits/structure/homologous_superfamily/cath/s-adenosylmethionine-decarboxylase-3-30-360-50.yaml:1-15`; `data/traits/structure/homologous_superfamily/cath/s-adenosylmethionine-decarboxylase-3-60-90-10.yaml:1-15`).

Using the stored `P17707` sequence as the shared UniProt frame, the stored PROSITE pattern matches residues 62-72 (`AYVLSESSMFV`). That interval was computed locally from the repository's `sequence_pattern` against the repository's stored `P17707` sequence.

This example shows the smallest useful alignment product. The two records already share an explicit exemplar protein and sequence, and the source pattern can be placed on that same sequence. The resulting edge is not a same-axis equivalence claim: it says the sequence-defined motif and structure-derived active/binding site occupy overlapping residues on `P17707`.

## 6. Open questions / risks

- Isoforms: `CanonicalExample.protein_id` allows isoform suffixes (`UniProtKB:P00517-2` pattern in the schema), but sequence signatures and PDB mappings often target canonical accessions. The builder must reject or explicitly map isoforms.
- Numbering mismatches: UniProt feature coordinates, InterPro match coordinates, PDB author numbering, PDB label numbering, and AlphaFold residue indices are not interchangeable. SIFTS is mandatory for PDB-derived BioLiP2, MetalPDB, M-CSA/PDB, CATH, ECOD, and SCOPe residue claims.
- Engineered constructs and missing residues: PDB structures can omit loops, truncate termini, mutate active-site residues, or include tags. Mapped residue sets need evidence of coverage, not just accession equality.
- Multi-chain complexes: BioLiP2 and M-CSA can describe ligand or catalytic sites assembled from multiple chains. A single UniProt frame is sufficient only for single-chain features; complex sites need chain-aware participant sets.
- Multi-domain proteins: same protein, same broad fold, and same family classification do not imply the same region. The existing localized-feature trap in `build_member_overlap.py` and `verify_region_overlap.py` applies directly.
- Aggregated class records: BioLiP2 and MetalPDB seeders aggregate source instances into class records. For alignment, the builder must recover the particular occurrence, not infer a residue set from the class label "ATP-binding site" or "dinuclear zinc site."
- Predicate choice: `biolink:overlaps`, `biolink:part_of`, `biolink:narrow_match`, and `biolink:related_to` are available shapes, but a curator may want a more precise RO predicate for "same physical residue/site by different evidence." Until then, put confidence and method in `relation_source`.
- Curator judgement: exact coordinate overlap is evidence of physical correspondence, not always biological equivalence. Ambiguous motifs, broad domains, weak SIFTS mappings, and partial overlaps should stay review-tier edges.
